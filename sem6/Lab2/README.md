# Лабораторная работа 10
## Методы оптимизации вычисления кода с помощью потоков и процессов

**Цель:** исследовать методы оптимизации вычисления кода, используя потоки и процессы на основе сравнения времени вычисления функции численного интегрирования методом левых прямоугольников.

---

## Итерация 1. Базовая реализация

Реализована функция `integrate()` с полным docstring по PEP 257, аннотациями типов (`Callable[[float], float]`), двумя doctest-примерами и четырьмя юнит-тестами.

Замеры времени выполнения (чистый Python):

| n_iter | Время |
|--------|-------|
| 1 000 | ~0.13 мс |
| 10 000 | ~1.37 мс |
| 100 000 | ~13.24 мс |
| 1 000 000 | ~131.39 мс |

Время растёт линейно с увеличением числа итераций.

---

## Итерация 2. Оптимизация с помощью потоков

Реализована функция `integrate_threads()` на основе `ThreadPoolExecutor`. Отрезок интегрирования делится между потоками равномерно.

Замеры времени (n_iter = 1 000 000):

| Потоков | Время |
|---------|-------|
| 1 | ~111.70 мс |
| 2 | ~108.97 мс |
| 4 | ~114.68 мс |
| 6 | ~113.61 мс |
| 8 | ~115.93 мс |

Увеличение числа потоков не даёт ускорения — из-за **GIL (Global Interpreter Lock)**. GIL не позволяет нескольким потокам одновременно выполнять байткод Python, поэтому для CPU-bound задач потоки работают поочерёдно. Время даже растёт из-за накладных расходов на создание потоков.

---

## Итерация 3. Оптимизация с помощью процессов

Реализована функция `integrate_processes()` на основе `multiprocessing.Pool`. Каждый процесс имеет собственный интерпретатор Python и не ограничен GIL.

Замеры времени (n_iter = 1 000 000):

| Процессов | Время |
|-----------|-------|
| 1 | ~251.86 мс |
| 2 | ~247.10 мс |
| 4 | ~250.84 мс |
| 6 | ~202.75 мс |
| 8 | ~179.04 мс |


---

## Вывод

Потоки не подходят для ускорения CPU-bound вычислений в Python из-за GIL. Процессы обходят это ограничение и дают реальный параллелизм, однако имеют накладные расходы на запуск и межпроцессное взаимодействие, которые окупаются только при достаточно большом объёме вычислений.
